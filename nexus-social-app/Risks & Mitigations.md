# Risks & Mitigations

## Overview
This document captures the primary risks identified for the Nexus Social Platform and the mitigations put in place or planned.

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| **Hard‑coded secrets** | High – could expose credentials | Low (now) | Replaced with `${process.env.VAR}` and added `.env.example`. Lint rule `no‑secrets` blocks new hard‑coded values. |
| **Unrestricted API rate** | Medium – abuse could degrade service | Medium | In‑memory rate limiting (100 req/min per IP) added to `src/middleware.ts`. |
| **Missing CORS headers** | Medium – cross‑origin requests may fail | Low | Centralised CORS handling via `handleCors` in middleware. |
| **Lack of security headers** | High – vulnerable to click‑jacking, MIME sniffing, XSS | Low | Standard security headers (Content‑Security‑Policy, X‑Frame‑Options, etc.) added in middleware. |
| **Insufficient CI artifact retention** | Low – artifacts could be lost before review | Low | GitHub Actions workflow uploads `artifacts/` and retains for 7 days. |
| **OpenTelemetry collector not started** | Medium – telemetry loss | Low | CI workflow explicitly starts and stops the OTEL collector via Docker Compose. |
| **Missing environment configuration** | High – runtime failures | Medium | `.env.example` provides a template; CI sets required secrets via GitHub Secrets. |

## Future Mitigations
- Implement persistent rate‑limiting backed by Redis for distributed deployments.
- Add automated secret‑scan step in CI (e.g., `trufflehog`).
- Enforce CSP nonce generation for dynamic scripts.
- Review and tighten Docker Compose network isolation for OTEL components.
